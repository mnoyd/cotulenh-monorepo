## Epic Summary

### Delivery Metrics

- Stories completed: 6/6 (100%)

---

## Successes & Strengths

1. **Server Action + Zod + useActionState pattern** — Established in Story 1-3, became the gold standard for all form submissions. Return shape `{ success, error? }` with Vietnamese Zod messages.

2. **Test infrastructure from day one** — vitest + Testing Library + co-located `__tests__/` pattern. Steady growth: 14 → 59 → 83 → 96 → 124 tests.

3. **PKCE password reset flow** — Story 1-4 implemented cleanly with zero debug issues. Auth callback Route Handler pattern well-established.

4. **Self-correction over time** — Review findings decreased story over story. Stories 1-5 and 1-6 had zero challenges in dev notes.

5. **Responsive app shell** — Sidebar (desktop ≥1024px) and bottom tab bar (mobile <1024px) with proper touch targets, active route detection, and keyboard accessibility.

---

## Challenges & Growth Areas

1. **Design tokens defined but not wired** — Story 1-1 defined CSS custom properties but didn't apply them to Tailwind/shadcn. Caught in review. Required explicit mapping fix.

2. **Framework defaults fighting design system** — shadcn re-added Geist fonts, button corners remained rounded despite 0px requirement. Each framework has opinions that must be actively overridden.

3. **Vietnamese text compliance** — English text leaked into Stories 1-1 and 1-2. Required active enforcement in every component, not just top-level pages.

4. **Story 1-3 placeholder submission** — Initial implementation was placeholder code, not full auth flow. Caught in review. Different model (GPT-5 Codex) interpreted story differently.

5. **Placeholder routes needed proactively** — Stories 1-2 and 1-5 required placeholder routes to avoid broken navigation during phased delivery.

---

## Key Insights & Learnings

- **Patterns established in early stories (1-1 through 1-3) paid dividends in later stories (1-4 through 1-6)**
- **Code review is the critical safety net** — especially for first epic when patterns aren't yet established
- **Server Components vs Client Components boundary requires deliberate design** — cookie writes, button variant exports, pathname detection all need clear component type decisions
- **Empty states with forward-path CTAs are good MVP practice** — dashboard sections ready for data injection
- **Skeleton screens only, never spinners** — consistent loading UX established
