## Successes & Strengths

1. **Learn system as acquisition funnel** — Core product bet shipped as designed. Visitors learn without signup, track progress locally, get nudged to create account after 3+ lessons. Growth loop working.

2. **useBoard hook + dynamic imports** — Story 2-2 bridged vanilla TS board library into React with imperative API, no remounts on config changes. Pattern carries forward to Epic 3 game board.

3. **Test quality matured** — Beyond happy-path: SSR hydration edge cases, localStorage unavailability, zero-star completions, auth state transitions. 128 new tests with real depth.

4. **Progressive hint system** — Pulse at 10s, arrow at 20s, text at 40s, respecting `prefers-reduced-motion`. Polished UX.

5. **Zustand + ProgressManager pattern scales** — Established in 2-1, extended cleanly through 2-4. External state manager sync with Zustand store is a reusable pattern.

6. **Merge strategy (keep best stars)** — Simple, correct, no data loss. localStorage stars vs DB stars, upsert only when local > DB.

---

## Challenges & Growth Areas

1. **Story 2-4 "done but not done"** — First review found 2 CRITICAL issues: login merge flow not implemented despite task marked complete, required test coverage missing. Second occurrence of this pattern (after Story 1-3).

2. **SSR/client hydration friction** — localStorage-backed state caused client/server mismatches in Story 2-3. Required explicit loading placeholders (`progressPending` flag). Recurring theme.

3. **Auth state transition complexity** — `loading → authenticated → migration` flow has subtle edge cases. `authState='loading'` could incorrectly trigger localStorage writes for authenticated sessions.

4. **N+1 DB reads in Story 2-4** — `flushPendingSaves` and `persistProgress` each calling `getDbProgress` separately. Caught in review, consolidated into single `hydrateFromDb` helper.

5. **SignupPrompt flash during auth loading** — Component briefly showed before auth state resolved. Required guards in both `learn-hub-client.tsx` and `lesson-view.tsx`.

---

## Key Insights & Learnings

- **"Done ≠ done" is a systemic risk** — Two stories across two epics had this problem. Need Definition of Done checklist.
- **Adversarial code review catches what first-pass misses** — Story 2-4's second review found the critical N+1 and auth loading issues.
- **SSR hydration requires a deliberate pattern** — Any component reading client-side state needs explicit loading placeholder. This should be a checklist item.
- **Performance patterns matter early** — N+1 DB reads in a learn system are forgivable; in realtime games they're fatal. Epic 3 needs performance awareness from Story 1.
- **Board integration pattern is proven** — `next/dynamic` with `ssr: false`, imperative API via ref, config updates in place. Ready for game mode.
